How to run the Berlin Secondary Sales Price API (full model)
==============================================================

Run all commands from the project root:
/home/fabian/code/MarieKatha/berlin-property-intelligence

Make sure the "berlin-property" pyenv env is active (it's the default for
this directory via .python-version). If commands aren't found, use the
full path instead, e.g. /home/fabian/.pyenv/versions/berlin-property/bin/uvicorn

This app serves the 18-feature model trained in MLlogic-sales/
-- it imports MLlogic-sales/predict.py
directly rather than duplicating any training or preprocessing logic.

1. (One-time, or after updating raw_data/secondary_sales.csv) Train the model:

   python MLlogic-sales/train.py

   This builds the full feature pipeline and saves MLlogic-sales/model.pkl.
   See MLlogic-sales/instructions.md for details.

2. Start the API:

   uvicorn api_sales.fast:app --port 8001

   (a different port than api/'s 8000, so both APIs can run side by side)

   - Visit http://127.0.0.1:8001/ in a browser for a welcome page with
     example links (minimal, minimal + listing_price, and a full listing).
   - Or query directly, e.g. the minimal required set:

     curl "http://127.0.0.1:8001/predict?ortsteil=Kreuzberg&area_m2=69&condition=saniert"

     ...or with listing_price added, to also get listing_vs_predicted_pct:

     curl "http://127.0.0.1:8001/predict?ortsteil=Kreuzberg&area_m2=69&condition=saniert&listing_price=600000"

   - Interactive docs: http://127.0.0.1:8001/docs
     (ortsteil, energy_class, condition and position all render as
     dropdowns there, since they're typed as Enums in fast.py)

Query parameters
-----------------

Required (missing any of these returns 422):

   ortsteil       -- dropdown: all 82 Berlin neighbourhoods in the training
                      data, e.g. Kreuzberg
   area_m2        -- living area in m^2
   condition      -- dropdown: renovierungsbeduerftig, renoviert, modernisiert,
                      saniert, kernsaniert

Optional (every field not listed here is filled with its training-set
average if omitted -- see the accuracy caveat below):

   listing_price  -- the listing's actual/asking price in EUR. Not a model
                      feature. If given, the response includes
                      listing_vs_predicted_pct (see "Response shape" below);
                      if omitted, the response explains how to get it instead
   rooms
   floor, total_floors
   year_built
   energy_class               -- dropdown: A_plus, A, B, C, D, E, F, G, H
   has_lift, has_balcony, has_cellar, has_parking
   transit_distance_min
   mortgage_rate_at_listing
   position                    -- dropdown: gartenhaus, hinterhaus, seitenfluegel, vorderhaus

Note: bezirk (Berlin district) and transit_line are NOT request parameters --
neither is a model feature. ortsteil's target encoding already captures
location more precisely than bezirk's coarser 12-district grouping, and
transit_line wasn't found useful enough to keep (see
MLlogic-sales/preprocessing.py).

Accuracy caveat
----------------

This API always fills missing optional fields with their training-set
average (MLlogic-sales/predict.py's fill_missing=True) rather than requiring
every field -- that's a genuine accuracy trade-off, not a free lunch. See the
notebook's "Robustness check" section (tested on its 35-feature model, before
bezirk/transit_line were dropped here): test MAE goes from ~€28,773 with
every feature known to ~€69,731 with only ortsteil/area_m2/condition known.
The same phenomenon applies to this leaner model -- the more optional fields
you actually provide, the more the prediction should be trusted.

Response shape:

   With listing_price:

     {"predicted_price_eur": 318784.16, "listing_vs_predicted_pct": 188.22,
      "message": "This offer is 88.22% more expensive than my prediction."}

     listing_vs_predicted_pct is listing_price / predicted_price_eur * 100.
     >100% means the listing is priced above the model's prediction
     (possibly overpriced) -- message says "X% more expensive than my
     prediction". <=100% means it's priced at or below it (possibly
     underpriced, or just a good deal) -- message says "X% cheaper than my
     prediction".

   Without listing_price:

     {"predicted_price_eur": 318784.16, "listing_vs_predicted_pct": null,
      "message": "if you specify the actual listing price I can compare it
      with my prediction and rate the value of the listing"}
